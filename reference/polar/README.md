# Polar AccessLink API ‚Äî Detailed Sport Info

> Reference documentation for Polar's exercise sport classification.

**Source:** [Polar AccessLink API ‚Äî detailed sport info values](https://www.polar.com/accesslink-api/#detailed-sport-info-values-in-exercise-entity)

## Versioning

The AccessLink API is versioned (`v3`, from the swagger `info.version`), but the
`detailed_sport_info` value list itself is **not separately versioned** ‚Äî it lives in
the HTML appendix, not the spec. We therefore pin `platform_version` in
`mappings/polar.yaml` to the API version plus the fetch date:
`"AccessLink API v3 (fetched 2026-06-08)"`. When refreshing, re-download `swagger.yaml`,
re-transcribe any new appendix values into `detailed_sport_info.yaml`, and bump the
fetch date.

## The two sport fields

A Polar exercise entity carries two sport-related fields:

- **`detailed_sport_info`** ‚Äî a string from the 174-value Polar Flow taxonomy
  (e.g. `ROAD_RUNNING`, `INDOOR_CYCLING`, `WATERSPORTS_KAYAKING`). **This is the
  mapping target.**
- **`sport`** ‚Äî a free-form string ("Sport name", example `OTHER`). The swagger types
  it as a bare `string` with no enumeration, so it carries no signal we can map
  against. It is **not** a mapping target; when `detailed_sport_info` is absent,
  callers get the `generic` decode fallback.

(In the XML schema the field is hyphenated `detailed-sport-info`; in the JSON schema it
is `detailed_sport_info`. Same field.)

## Overview

- **174 string values**, far broader than OST's taxonomy. Most have no OST equivalent
  (ball sports, martial arts, dance, motorsports, water sports, gym classes, etc.) and
  map to `null` (decode ‚Üí `generic`), exactly as Strava handles `AlpineSki`.
- **Modifiers as distinct values** ‚Äî Polar expresses several OST modifiers as separate
  sport strings: `INDOOR_CYCLING`/`SPINNING` ‚Üí `cycling+stationary`, `TREADMILL_RUNNING`
  ‚Üí `running+stationary`, `INDOOR_ROWING` ‚Üí `rowing+stationary`, `E_BIKE` ‚Üí
  `cycling+assisted`, `ROLLER_SKIING_*` ‚Üí `xc_skiing.*+roller`.
- **Finer than Polar's own FIT export.** Polar's published "sport type mapping in FIT
  files" sends several of these to FIT `GENERIC` (spinning, roller skiing); OST keeps
  the sport and the modifier. We consult that FIT table as an audit aid but map to OST.

## Why swagger.yaml does not drive the build

`swagger.yaml` is stored for provenance and API reference, but it **does not enumerate**
the sport values ‚Äî `detailed-sport-info` is declared `type: string`, `example: RUNNING`.
The enumeration is therefore hand-curated in `detailed_sport_info.yaml`, transcribed
from the HTML appendix. No build step parses `swagger.yaml`.

## Files

- [`swagger.yaml`](swagger.yaml) ‚Äî full AccessLink OpenAPI 3.0.0 spec (v3), as
  published at https://www.polar.com/accesslink-api/swagger.yaml. Reference only.
- `detailed_sport_info.yaml` ‚Äî hand-curated `{value, flow_label}` for all 174 values.
  **Source of truth.** (Added during implementation; see `plans/019-polar-mapping.md`.)
- `targets.yaml` ‚Äî generated by `scripts/build_reference/polar.py` from
  `detailed_sport_info.yaml`; the flat list of legal `detailed_sport_info` targets. Do
  not edit by hand. (Added during implementation.)
