# Plan: Add Suunto Activities Mapping

## Context

Suunto classifies each move with an integer **activity ID** (0–121), documented in
`reference/suunto/Activities.pdf` alongside Suunto's own mapping to FIT
`sport`/`sub_sport`. This plan adds `suunto` as a seventh OST platform.

Structurally Suunto is closest to **Wahoo**: a flat **integer** target with gaps in the
numbering (ID 89 is unassigned), an integer encode-fallback, `generic` decode-fallback,
and **no `target_coarsening`**. Wahoo is the line-for-line template for the mechanical
steps. The baseline is green (0.7.0, six platforms merged), so no prerequisite fix is
needed.

## Target shape and the two columns

The PDF has two Suunto columns and two FIT columns:

- **`Sports/Activity name`** + **`ID for the sport`** — Suunto's side. The **integer ID
  is the mapping target.** The name is not unique (IDs 4–9 are all "Sports"), so it
  cannot be the key; it is kept only as the row label/comment.
- **`Sport id` / `Sub-sport name` / `Sub-sport id`** — Suunto's *own* export to Garmin
  FIT. We **learn from** this (it reveals how Suunto conceives each activity's
  discipline/indoor/assist character) but **do not use it** — our target is the Suunto
  ID and our value is the OST sport.

Target type is `int`. `generate.py` emits the tables generically for any registered
scalar-target platform — **no codegen changes** (proven with Wahoo).

## Reference data

`reference/suunto/`:

- `Activities.pdf` — **already present.** The source document; the only authoritative
  artifact Suunto publishes for this list (no machine-readable enum, no swagger for the
  activity IDs).
- `README.md` — **written with this plan.** Source, versioning, why the integer ID is
  the target, the learn-from-FIT note.
- `activities.yaml` — hand-curated `cases:` of `{value, name}` for all 121 activities,
  transcribed from the PDF (text layer extracted with `pdftotext -layout`). Source of
  truth.
- `targets.yaml` — generated by `scripts/build_reference/suunto.py`; the flat list of
  legal activity IDs. Do not edit by hand.

`platform_version` pins the source + fetch date, since the PDF carries no version
number: `"Activities reference PDF (fetched 2026-06-08)"`. (The newest activities,
103–121, are annotated "since March 2025", so the bundled list is ≥ 2025-03.)

## Learning from Suunto's FIT export — without copying it

As with Polar, OST is frequently *finer* than Suunto's own FIT export, and in a few
places we deliberately diverge:

| Suunto activity (ID) | Suunto → FIT (their export) | OST mapping (ours) | Why ours differs |
|---|---|---|---|
| E-mtb (106) | `E_BIKING` / E_BIKE_MOUNTAIN | `cycling.mountain+assisted` | OST keeps the mountain discipline **and** e-assist (two axes) instead of one e-bike sport. |
| Roller skiing (56) | `CROSS_COUNTRY_SKIING` | `xc_skiing+roller` | FIT has no roller concept; OST's `roller` modifier captures it. |
| E-bike (105) | `E_BIKING` / E_BIKE_FITNESS | `cycling+assisted` | e-assist is a modifier on cycling, not a separate sport. |
| Gravel cycling (99) | `CYCLING` / GRAVEL_CYCLING | `cycling.gravel` | OST's native discipline. |
| Hand cycling (109) | `CYCLING` / HAND_CYCLING | `cycling` | OST has no hand-cycling code; decode to nearest (consistent with Wahoo/Polar). |
| Telemark skiing (84) | `ALPINE_SKIING` | `null` | **Divergence:** OST models only `xc_skiing`; alpine touring has no code. |

Where Suunto's FIT classification is itself informative we follow it: **Backcountry
skiing (107)** is filed by Suunto under `CROSS_COUNTRY_SKIING`, so we map it to
`xc_skiing` (decode-only) rather than guessing alpine. The other touring activities
(Ski touring 31, Ski mountaineering 116, Splitboarding 110, Telemark 84) Suunto files
under `ALPINE_SKIING`/`MOUNTAINEERING`/`SNOWBOARDING` → `null`.

The FIT column also confirms several modifier assignments (treadmill/indoor →
stationary, gravel/cyclocross/mountain disciplines) and flags the long tail Suunto
itself sends to FIT `GENERIC` / `TRAINING` — a strong signal those belong as OST
`null`.

### Fallbacks

- `fallback.encode: 4` — unknown OST sport encodes to Suunto "Sports" (the generic slot).
- `fallback.decode: generic` — the preferred sport of ID 4, so it round-trips (rule 12).

### Everything else → `sport: null` (~89 values)

Suunto's taxonomy covers far more than OST models. All `null` (decode → `generic`):

- **Ball / team / racket sports** — soccer, tennis, basketball, badminton, baseball,
  volleyball, American football, table tennis, racquetball, squash, floorball,
  handball, softball, bowling, cricket, rugby, ice hockey, futsal, field hockey, padel.
- **Martial / combat** — combat sport, boxing.
- **Gym / strength / fitness** — indoor sports, outdoor gym, gym, fitness class,
  crossfit, crosstrainer (elliptical), stretching, kettlebell, aerobics, circuit
  training, cheerleading, gymnastics, calisthenics, jump rope, chores, yoga/pilates,
  pilates, yoga, meditation. *(See judgment call — only the literal "Sports" slots are
  `generic`; named fitness activities are `null`, matching the Polar convention.)*
- **Snow / alpine** — downhill skiing, ski touring, snowboarding, telemark,
  splitboarding, ski mountaineering, snow shoeing, biathlon. *(Backcountry skiing (107)
  is `xc_skiing`, not `null` — Suunto files it under CROSS_COUNTRY_SKIING.)*
- **Skating / wheels** — roller skating, skateboarding, ice skating.
- **Water** — paddling, water sports, sailing, kayaking, canoeing, standup paddling,
  openwater… (mapped), windsurfing, kitesurfing, surfing, snorkeling, scubadiving,
  freediving, mermaiding, spearfishing, fishing.
- **Air / outdoor** — paragliding, horseback riding, motorsports, golfing, frisbee
  golf, climbing, mountaineering, hunting, parkouring, dancing, ball games, esports n/a.
- **Multisport containers** — multisport, triathlon, swimrun, duathlon, aquathlon,
  adventure racing, transition (only individual legs would map, and Suunto exposes
  none).
- **Other** — wheelchair sport, obstacle racing, orienteering.

### Judgment calls (maintainer review)

- **Named gym/fitness → `null`, not `generic`.** Only the literal "Sports" slots (4–9)
  are `generic` (Suunto's explicit unspecified bucket); crossfit, gym, kettlebell,
  elliptical, etc. are `null`. This follows the **Polar** convention. *(Note: Wahoo
  mapped its `FE_*`/`WORKOUT` fitness-equipment types to `generic` per an explicit
  earlier decision; Suunto/Polar treat named fitness activities as `null`. Both decode
  to `generic` at runtime — the difference is "classified as generic" vs "no
  equivalent". Flagged so the maintainer can align the convention if desired.)*
- **`Telemark` (84), `Ski touring` (31), `Splitboarding` (110), `Ski mountaineering`
  (116) → `null`** — the alpine-touring/snowboard family OST has no code for. (But
  **`Backcountry skiing` (107) → `xc_skiing`**, following Suunto's own FIT
  classification, which files it under `CROSS_COUNTRY_SKIING`.)
- **`Orienteering` (60) → `null`** — a distinct navigation sport; the running is
  incidental. Consistent with plain `ORIENTEERING` in the Polar mapping.
- **`Snorkeling` (90) → `null`** (Suunto FIT → `SWIMMING`) — snorkeling is not the
  swimming sport.
- **`Vertical running` (115) → `running.trail`** and **`Nordic walking` (24) →
  `walking`** — in-family but lossy; kept as nearest standard sport.

## Validation dry-run

Checked against `scripts/generate.py` rules before writing:

- **Rule 8 (one preferred per sport):** each of the 22 mapped OST sports/combos has
  exactly one preferred row; the multi-synonym sports (`generic` ×6, `walking` ×2,
  `walking.hiking` ×2, `running.track` ×2, `running.trail` ×2, `cycling` ×2,
  `xc_skiing` ×2) each name one preferred.
- **Rules 10–11 (round-trip):** every preferred entry round-trips
  (`cycling.mountain+assisted` → 106, `xc_skiing+roller` → 56, etc.); every `null` row
  decodes to `generic`.
- **Rule 12 (fallback round-trip):** `fallback.decode: generic` is the sport of
  preferred ID 4.

No violations found; first implementation step runs the real `make generate && make
lint && make test`.

## Verification

```python
from open_sport_taxonomy import Sport
from open_sport_taxonomy.platforms import suunto

suunto.encode(Sport("cycling.gravel"))                 # 99
suunto.encode(Sport("cycling.mountain+assisted"))      # 106 (discipline + assist)
suunto.encode(Sport("xc_skiing+roller"))               # 56  (finer than FIT GENERIC)
suunto.encode(Sport("cycling.road"))                   # 2   (no Suunto road type → cycling)
suunto.encode(Sport("running.road+commute"))           # 1   (no road; +commute dropped)
suunto.decode(52)                                       # cycling+stationary
suunto.decode(109)                                      # cycling (hand cycling)
suunto.decode(84)                                       # generic (telemark → null)
suunto.decode(999)                                      # generic (unknown → fallback)
```

## Complexity

**Low–Moderate**, identical in shape to Wahoo. Flat int target, zero `generate.py`
codegen, `wahoo` as the template. The non-trivial parts: hand-curating
`activities.yaml` (121 rows from the PDF) and the mapping judgment (heavy `null` tail +
the alpine-skiing and gym-vs-generic calls). Same mechanical touch-points as every
platform. No architectural changes.

## Decisions

1. **Target = Suunto activity ID (integer)**; the activity name is a label only.
2. **`Hand cycling → cycling`**, consistent with Wahoo/Polar.
3. **`platform_version = "Activities reference PDF (fetched 2026-06-08)"`** — pin
   source + fetch date (the PDF is unversioned).

4. **Named gym/fitness → `null`** (gym, crossfit, kettlebell, elliptical, aerobics, …).
   Resolved: follow the Polar convention (named fitness = no OST equivalent); only the
   literal "Sports" slots are `generic`.
5. **`Backcountry skiing` (107) → `xc_skiing`** (decode-only). Resolved: follow Suunto's
   own FIT classification (`CROSS_COUNTRY_SKIING`).
