# Python Reference Implementation

> A lightweight Python library for working with OpenSportTaxonomy codes, modifiers, and platform mappings.

**Status:** Draft
**Package name:** `open-sport-taxonomy`
**Import:** `import open_sport_taxonomy`

---

## How a developer uses this

A developer building a sports data application (e.g. a fitness dashboard that ingests from Strava and exports to Apple Health) needs to:

1. **Represent activities** using a canonical sport code + modifiers
2. **Parse and serialize** sport strings (`"cycling.road+race+virtual"`)
3. **Translate outbound data** to a platform value (e.g. HealthKit `13`, Strava `"GravelRide"`)
4. **Navigate the taxonomy** — find all cycling disciplines, get a sport's parent
5. **Validate** that a sport code exists and modifiers are legal

The library should make all of these one-liners.

## Proposed interface

### Sport — the primary type

`Sport` is a frozen dataclass wrapping a sport code and modifiers. It is the single type a developer works with. The constructor parses encoded strings (see plan 006), validates against the taxonomy, and checks modifier constraints.

The two dataclass fields are `code` (str) and `modifiers` (frozenset). Taxonomy attributes (`label`, `parent`, `disciplines`) are properties that look up from an internal index — they are not stored on the instance.

```python
from open_sport_taxonomy import Sport, Modifier

# Construct from encoded string (primary entry point for external data)
sport = Sport("cycling.road+race+virtual")
sport = Sport("cycling.road")

# Construct with explicit modifiers
sport = Sport("cycling.road", modifiers={Modifier.RACE, Modifier.VIRTUAL})

# Class constants — IDE autocomplete, hardcoded references
Sport.CYCLING_ROAD                  # Sport("cycling.road"), no modifiers
Sport.CYCLING                       # Sport("cycling"), no modifiers
Sport.RUNNING_TRAIL                 # Sport("running.trail"), no modifiers
```

#### Fields (stored on instance, used for equality and hashing)

```python
sport = Sport("cycling.road+race+virtual")

sport.code                          # "cycling.road"
sport.modifiers                     # frozenset({Modifier.RACE, Modifier.VIRTUAL})
```

#### Properties (derived from taxonomy, not stored)

```python
sport.label                         # "road cycling" (English display name)
sport.parent                        # Sport("cycling") — parent sport, no modifiers
sport.disciplines                   # () — direct child sports (empty for a discipline)

Sport.CYCLING.parent                # None (top-level sport)
Sport.CYCLING.disciplines           # (Sport("cycling.cyclocross"), Sport("cycling.gravel"), ...)
```

`parent` and `disciplines` always return Sport instances without modifiers — the taxonomy hierarchy is about the sport code, not modifiers.

#### String representation

```python
str(sport)                          # "cycling.road+race+virtual" (canonical encoded form)
str(Sport.CYCLING_ROAD)             # "cycling.road"
repr(sport)                         # "Sport('cycling.road+race+virtual')"

# Roundtrip
Sport(str(sport)) == sport          # True
```

#### Equality and hashing

Equality and hashing are based on the two fields: `code` and `modifiers`. These are generated by the frozen dataclass — not hand-written.

```python
Sport("cycling.road") == Sport.CYCLING_ROAD                 # True
Sport("cycling.road+virtual") != Sport("cycling.road")      # True
{Sport.CYCLING_ROAD: "some value"}                          # works as dict key
```

#### Validation

```python
Sport("cycling.fake")               # ValueError: Unknown sport code: 'cycling.fake'
Sport("cycling.road+race+commute")  # ValueError: Modifiers 'race' and 'commute' conflict (same group: 'purpose')
Sport("cycling.road+unknown")       # ValueError: Unknown modifier: 'unknown'
Sport("cycling.road+")              # ValueError: Invalid encoded string
Sport("")                           # ValueError: Invalid encoded string
```

#### Listing and filtering

```python
Sport.all()                         # list of all sports in the taxonomy (no modifiers)
Sport.CYCLING.disciplines           # direct children only
```

For deeper filtering, use standard Python on the flat list:

```python
[s for s in Sport.all() if s.code.startswith("cycling.")]
```

### Modifier — standard Enum

`Modifier` is a small fixed set with no composite behavior — a natural fit for `Enum`.

```python
from open_sport_taxonomy import Modifier

Modifier.VIRTUAL                    # <Modifier.VIRTUAL: 'virtual'>
Modifier.VIRTUAL.code               # "virtual" (same as .value, provided for domain consistency)
Modifier.VIRTUAL.label              # "virtual"
Modifier.VIRTUAL.group              # None (independent flag)

Modifier.RACE.group                 # "purpose"

Modifier("virtual")                 # Modifier.VIRTUAL (lookup by value)
Modifier("unknown")                 # ValueError

# Group filtering
Modifier.in_group("purpose")        # [Modifier.COMMUTE, Modifier.RACE, Modifier.TRAINING]

# Iteration
list(Modifier)                      # [Modifier.ASSISTED, Modifier.COMMUTE, ...]
```

### Platform translation (outbound)

```python
from open_sport_taxonomy.platforms import strava, apple_healthkit, garmin_fit

# Translate a Sport (with or without modifiers)
strava.translate(Sport.CYCLING_GRAVEL)                      # "GravelRide"
strava.translate(Sport("cycling.road+virtual"))              # "VirtualRide"
apple_healthkit.translate(Sport.CYCLING_ROAD)                # 13
garmin_fit.translate(Sport.CYCLING_ROAD)                     # GarminFitCode(sport=2, sub_sport=7)

# Fallback chain: exact match -> without modifiers -> parent code -> platform fallback
strava.translate(Sport("cycling.road+stationary"))           # "Ride"
```

### Schema metadata

```python
import open_sport_taxonomy
open_sport_taxonomy.version              # "0.1.0"
```

## Implementation

### Build-time code generation

The Python source files are generated from `schema.yaml` and `mappings/*.yaml` by `scripts/generate.py`. The YAML files are the source of truth; the generated `.py` files are committed to the repo (same pattern as `dist/reference.md`). No YAML is parsed at runtime. No runtime dependencies.

```
schema.yaml  ─┐
mappings/*.yaml ─┤── scripts/generate.py ──▶  _modifier.py   (Modifier enum + validation)
               │                              _sport.py       (Sport class + taxonomy data + constants)
               └──────────────────────────▶  _platforms.py   (mapping dicts for all platforms)
```

Run `uv run scripts/generate.py` when the schema or mappings change. Run `uv run scripts/generate.py --check` in CI to verify generated files are in sync.

### What is generated

**`_modifier.py`** — a real `class Modifier(str, Enum)` with hardcoded members, label/group lookup dicts, and `validate_modifiers()`. Type checkers see every member natively.

**`_sport.py`** — taxonomy data as plain Python dicts (`_LABELS`, `_PARENTS`, `_CHILDREN`), the `Sport` frozen dataclass, and class constant assignments (`Sport.CYCLING_ROAD = Sport("cycling.road")`). Type checkers see every constant via `# type: ignore[attr-defined]` annotations.

**`_platforms.py`** — pre-built mapping dicts and fallback values for all platforms, using the same `(code, frozenset)` key format that `Platform.translate()` expects. Garmin FIT targets are `GarminFitCode` namedtuples.

### What is hand-written

**`_platform.py`** — the `Platform` class (translate logic, fallback chain) and `GarminFitCode` namedtuple. Pure behavior, no data.

**`platforms/_strava.py`**, **`_apple_healthkit.py`**, **`_garmin_fit.py`** — one-liner modules that instantiate `Platform` with the generated data.

**`platforms/__init__.py`** — re-exports the three platform instances.

**`__init__.py`** — re-exports `Sport`, `Modifier`, `GarminFitCode`, `version`.

### Sport: frozen dataclass with `init=False`

`@dataclass(frozen=True, init=False, slots=True)` gives us:

| Decorator arg | What it gives us |
|---|---|
| `frozen=True` | True immutability (`sport.code = "x"` raises `FrozenInstanceError`). Generates `__eq__` and `__hash__` based on `(code, modifiers)`. |
| `init=False` | Custom `__init__` that parses encoded strings and validates. The auto-generated `__init__` would have the wrong signature. |
| `slots=True` | Memory efficiency. Prevents adding arbitrary attributes. |

`object.__setattr__` in `__init__` is the documented pattern for setting fields on a frozen dataclass with `init=False`.

**What we get for free:** `__eq__`, `__hash__`.
**What we write:** `__init__`, `__str__`, `__repr__`, properties.

The constructor accepts `str` only — not `str | Sport`. This avoids ambiguous overloading. Three construction forms, all from strings:

```python
Sport("cycling.road")                                       # bare code
Sport("cycling.road+virtual")                               # encoded string
Sport("cycling.road", modifiers={Modifier.VIRTUAL})         # code + explicit modifiers
```

The encoded string and explicit modifiers are mutually exclusive — passing both raises `ValueError`.

### Modifier: `str` Enum

`Modifier(str, Enum)` with hardcoded members. Because it extends `str`, modifier values compare directly with strings and work in f-strings without `.value`:

```python
Modifier.VIRTUAL == "virtual"       # True
f"modifier: {Modifier.VIRTUAL}"     # "modifier: virtual"
```

### Platform

The `Platform` class implements the matching rules from plan 004. `GarminFitCode` is a `NamedTuple` for typed attribute access (`result.sport`, `result.sub_sport`).

## Design decisions

### Build-time generation, not runtime loading

The generated Python files are committed to the repo. The YAML is parsed only by the generator script, never at runtime. This means:

- **Zero runtime dependencies** — no `pyyaml` for consumers
- **Instant import** — no YAML parsing, no dynamic enum construction
- **Type checkers see everything natively** — real Enum members, real class constants, no `.pyi` stubs needed
- **What ships is what you inspect** — a developer reading the source sees real Python

The trade-off is a manual step when the schema changes (`uv run scripts/generate.py`). This is the same workflow the project already uses for `dist/reference.md` via `generate_reference.py`.

### Sport as a frozen dataclass, not an Enum

Sport started as an Enum but evolved into a frozen dataclass because:

- **The constructor must parse encoded strings** — `Sport("cycling.road+virtual")` should just work. Enum constructors only do value lookup; they can't parse and split.
- **Sport carries modifiers** — an Enum member is a single fixed value. A Sport is a (code, modifiers) pair where the modifier set varies.
- **Frozen dataclass gives us the best of both worlds** — `__eq__` and `__hash__` for free (like Enum), true immutability (like Enum), plus a custom constructor.

Class-level constants (`Sport.CYCLING_ROAD`) preserve the autocomplete and discoverability that Enums provide.

### Modifier stays an Enum

Modifiers are a small fixed set (6 members) with no composite behavior. A real Enum is the right fit — full iteration, membership, identity, and no custom constructor needed. Using `str` as the mixin type makes modifiers interoperable with plain strings.

### Constructor accepts str only

The Sport constructor takes a `str`, not `str | Sport`. This avoids ambiguous overloading: if `Sport(existing_sport, modifiers={...})` were allowed, the semantics when the source already has modifiers would be unclear (merge? replace? error?). A single input type means one clear code path.

### No separate encode/decode

`Sport()` is the decoder. `str()` is the encoder. No standalone functions — the type itself handles serialization. One way in, one way out.

### Properties, not fields, for taxonomy data

Only `code` and `modifiers` are dataclass fields (stored on the instance, used for equality and hashing). `label`, `parent`, and `disciplines` are properties that look up from generated dicts. This separation means:

- Two Sport instances with the same code and modifiers are always equal
- The Sport instance stays small (two fields)
- Taxonomy data is not duplicated across instances

### ValueError for all construction errors

Invalid sport codes, unknown modifiers, and modifier group conflicts all raise `ValueError`. This is consistent with Python conventions — `int("abc")`, `Enum("invalid")`, and `ipaddress.IPv4Address("999.0.0.0")` all raise `ValueError`. A custom exception hierarchy adds API surface without practical benefit.

### GarminFitCode as NamedTuple

Garmin FIT maps to a (sport, sub_sport) pair. Returning a raw dict would be un-Pythonic — `NamedTuple` gives named attribute access, tuple unpacking, and a clean repr.

### Versioning

The library version tracks the schema version. `open_sport_taxonomy.version` returns the version from `schema.yaml`. The PyPI package version matches.

## Matching rules (from plan 004)

The `translate()` method implements the recommended fallback chain:

1. **Exact match** on sport code + modifiers set
2. **Drop modifiers** — match on sport code alone
3. **Walk up to parent** — try parent sport (e.g. `cycling.road` -> `cycling`)
4. **Fallback** — return the platform's generic/catch-all value

This is deterministic and always returns a value.

## Testing strategy

- **Construction** — from encoded strings, bare codes, code + explicit modifiers
- **String roundtrip** — `Sport(str(sport)) == sport` for every sport, with and without modifiers
- **Canonical ordering** — `str()` always produces alphabetically sorted modifiers
- **Validation** — unknown sport codes, unknown modifiers, modifier group conflicts, malformed input (empty, trailing `+`, double `+`, encoded string with explicit modifiers)
- **Equality and hashing** — value equality, dict keys, sets
- **Taxonomy navigation** — `label`, `parent`, `disciplines`, `Sport.all()`
- **Modifier enum** — iteration, membership, group filtering, string interop
- **Fallback chain** — verify each step of the platform matching rules
- **Exhaustive platform coverage** — every sport code translates to every platform without error
