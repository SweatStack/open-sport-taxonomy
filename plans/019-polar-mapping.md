# Plan: Add Polar AccessLink API Mapping

## Context

Polar's [AccessLink API](https://www.polar.com/accesslink-api/) classifies each exercise
with a string field, `detailed_sport_info`, drawn from a 174-value Polar Flow sport
taxonomy. This plan adds `polar` as a sixth OST platform.

Structurally Polar is closest to **Strava**: a flat **string** target, a string
encode-fallback, `generic` decode-fallback, and **no `target_coarsening`**. Strava is
the line-for-line template for the mechanical steps. Contrast with `apple_healthkit` /
`wahoo` (integer targets) and `garmin_fit` (struct target).

This follows `CONTRIBUTING.md` → "Adding a new platform" and the format-v3 spec in
`docs/translation.md`. The Wahoo addition (plan 018) is the proven precedent; this plan
reuses its hard-won accuracy about the generator and test wiring.

## Prerequisite: green baseline

This plan assumes the `schema.yaml`/`pyproject.toml` version-consistency fix from the
Wahoo work (commit on `add-wahoo-mapping`) is in the base branch. If Polar lands
independently of Wahoo, first confirm `make lint` is green; if it fails with a version
mismatch, bump `schema.yaml` to match `pyproject.toml` and `make generate` before
starting. (See plan 018 "Prerequisite" for the full story.)

## Target shape and the two sport fields

The Polar exercise entity has **two** sport-related fields:

- **`detailed_sport_info`** — a string from the 174-value enumeration
  (e.g. `ROAD_RUNNING`, `INDOOR_CYCLING`, `WATERSPORTS_KAYAKING`). **This is the
  mapping target.**
- **`sport`** — a free-form string ("Sport name", example `OTHER`). The swagger
  declares it `type: string` with no enumeration. It is **not** a mapping target — the
  same reasoning that excluded Wahoo's derived family/location: it carries no
  enumerable signal we can map against. When `detailed_sport_info` is absent on an
  exercise, callers get `fallback.decode` (`generic`); we do not attempt to parse the
  free-form `sport` string.

Target type is `str`, like Strava. `generate.py` emits the tables generically for any
registered string-target platform — **no codegen changes** (verified against the
current generator during the Wahoo work).

## Reference data: swagger present, enum hand-curated

A subtlety worth stating plainly: **`reference/polar/swagger.yaml` (already stored)
does not contain the sport enumeration.** The OpenAPI spec types `detailed-sport-info`
as a bare `string` with `example: RUNNING`. The 174 values live only in the HTML
appendix ([detailed sport info values](https://www.polar.com/accesslink-api/#detailed-sport-info-values-in-exercise-entity)).

So `reference/polar/` holds:

- `swagger.yaml` — **already stored** (this change). The full AccessLink OpenAPI 3.0.0
  spec (v3), kept for API reference and provenance; not parsed by any build step.
- `detailed_sport_info.yaml` — **hand-curated source of truth**, one entry per value
  with `value` and `flow_label` (the human-readable "Sport in Polar Flow" name),
  transcribed from the appendix. This is the genuinely new manual work.
- `targets.yaml` — generated by `scripts/build_reference/polar.py`; the flat list of
  legal `detailed_sport_info` strings. Do not edit by hand.

`platform_version` tracks the swagger's API version plus the fetch date, since the
sport list itself is unversioned: `"AccessLink API v3 (fetched 2026-06-08)"`.

## Learning from Polar's FIT export — without copying it

Polar publishes its own ["sport type mapping in FIT files"](https://www.polar.com/accesslink-api/#sport-type-mapping-in-fit-files)
(Polar Flow label → FIT `sport`/`sub_sport`). We **consult it as an audit aid** — it
reveals how Polar conceives each sport's indoor/technique/assist character — but we map
to **OST**, not to FIT, and OST is frequently *finer* than Polar's own FIT export:

| Polar value | Polar → FIT (their export) | OST mapping (ours) | Why ours differs |
|---|---|---|---|
| `SPINNING` | `GENERIC` / INDOOR_CYCLING | `cycling+stationary` | FIT drops the sport to generic; OST keeps cycling + the modifier. |
| `ROLLER_SKIING_CLASSIC` | `GENERIC` | `xc_skiing.classic+roller` | FIT has no roller concept; OST's `roller` modifier captures it. |
| `ROLLER_SKIING_FREESTYLE` | `GENERIC` | `xc_skiing.skate+roller` | Same; plus freestyle = skate technique. |
| `E_BIKE` | `E_BIKING` (distinct sport) | `cycling+assisted` | OST models e-assist as a modifier on cycling, not a separate sport. |
| `XC_SKIING_FREESTYLE` | `CROSS_COUNTRY_SKIING` | `xc_skiing.skate` | FIT loses the technique; OST keeps skate vs classic. |
| `GRAVEL` | `CYCLING` / GRAVEL_CYCLING | `cycling.gravel` | Same intent, OST's native discipline. |

The FIT table also confirms several modifier assignments (treadmill/indoor → stationary,
gravel/road/mountain sub-disciplines), and flags the long tail Polar itself sends to
`GENERIC` — a strong signal those belong as OST `null` (no equivalent).

designed escape hatch (decode → `generic`), exactly as Strava handles `AlpineSki`.
`preferred: true` marks the single encode target per OST sport.

### Fallbacks

- `fallback.encode: OTHER_OUTDOOR` — unknown OST sport encodes to Polar's outdoor
  catch-all (Polar has no single "OTHER"; only `OTHER_INDOOR`/`OTHER_OUTDOOR`).
- `fallback.decode: generic` — the preferred sport of `OTHER_OUTDOOR`, so it
  round-trips (rule 12). `encode(generic) == OTHER_OUTDOOR`, `decode(OTHER_OUTDOOR) ==
  generic`.

### Everything else → `sport: null` (~130 values)

Polar's taxonomy covers a vast space OST deliberately does not model. All of these are
`null` (decode → `generic`):

- **Ball / team sports** — soccer, basketball, volleyball, tennis, padel, cricket,
  rugby, handball, hockey variants, baseball, floorball, futsal, ultimate, etc.
- **Martial arts / combat** — boxing, judo, kickboxing, taekwondo, fitness martial arts.
- **Dance** — all `*_DANCING`, ballet, ballroom, latin, modern, show, street, jazz.
- **Gym / strength / mobility** — strength training, kettlebell, calisthenics, core,
  circuit, HIIT, bootcamp, functional training, cross-trainer, all `LES_MILLS_*`, yoga,
  pilates, stretching, mobility, step, stair workouts.
- **Snow / alpine** — downhill, telemark, backcountry skiing, snowboarding, biathlon,
  snowshoe trekking (OST models only `xc_skiing`).
- **Skating** — ice, inline, roller blading, skateboarding, skating.
- **Water sports** — canoeing, kayaking, surfing, sailing, windsurfing, kitesurfing,
  SUP, wakeboarding, water skiing, water running, aqua fitness.
- **Motorsports** — all `MOTORSPORTS_*`.
- **Parasports** — sled hockey, wheelchair racing/basketball/tennis, adaptive water
  skiing (except hand cycling → `cycling`, above).
- **Multisport containers** — `TRIATHLON`, `DUATHLON`, `OFFROAD*` (the events
  themselves), `FREE_MULTISPORT` (only the individual legs map).
- **Other** — golf, disc golf, riding, trotting, climbing/vertical, esports, agility,
  shooting, obstacle course racing, kickbike, skierg, water exercise.

### Judgment calls (maintainer review)

- **`KICKBIKE` → `null`** (Polar's FIT export says `CYCLING`). A kick scooter is not a
  cycling discipline in OST's sense; `null` over Polar's coarser choice.
- **`CROSS_COUNTRY_RUNNING` → `running.trail`**, **`RUCKING` / `NORDIC_WALKING` →
  `walking`** — in-family but lossy; the alternative is `null`. Kept as the nearest
  standard sport since the activity is genuinely the same.
- **`BIATHLON`, `OBSTACLE_COURSE_RACING`, `SNOWSHOE_TREKKING` → `null`** — distinct
  sports whose locomotion overlaps an OST code but which would lose essential meaning
  if flattened. `null` (→ `generic`) is the honest choice.
- **`ORIENTEERING_SKI` / `ORIENTEERING_MTB` → `xc_skiing` / `cycling.mountain`** — here
  we *do* map (decode-only), because the recorded activity is dominated by the
  locomotion. Borderline against the biathlon call; flagged for review.
- **`generic → OTHER_OUTDOOR`** (vs `OTHER_INDOOR`) — Polar forces an indoor/outdoor
  split with no neutral "other"; outdoor is the less-surprising default.

## Validation dry-run

Checked against `scripts/generate.py` rules before writing:

- **Rule 8 (one preferred per sport):** each of the 24 mapped OST sports/combos has
  exactly one preferred row; multi-synonym sports (`running` ×5, `running.trail` ×4,
  `cycling` ×4, `cycling.mountain` ×3, `swimming.open_water` ×3, `walking` ×3,
  `cycling+stationary` ×2, `xc_skiing` ×2, `generic` ×2) each name one preferred.
- **Rules 10–11 (round-trip):** every preferred entry round-trips; every `null` row
  decodes to `generic` via fallback.
- **Rule 12 (fallback round-trip):** `fallback.decode: generic` is the sport of
  preferred `OTHER_OUTDOOR`.

No violations found; `make generate` / `make lint` should pass once implemented. (Hand
trace, not executed — first implementation step runs the real `make generate && make
lint && make test`.)

## Complexity

**Low–Moderate**, comparable to Wahoo. Flat string target, zero `generate.py` codegen,
`strava` as the template. The two non-trivial parts:

1. **Hand-curating `detailed_sport_info.yaml`** — 174 values transcribed from the HTML
   appendix (the swagger doesn't enumerate them). Larger than Wahoo's 61, but
   mechanical.
2. **The mapping judgment** — heavier tail of `null`s than any prior platform, plus a
   dozen genuine borderline calls (documented above). The modifier-bearing rows are
   settled by existing conventions and are where OST visibly beats Polar's own FIT
   export.

Same six mechanical touch-points as every platform (3 registration sites, runtime
instance, `__init__.py`, 2 more test files) + docs/release hygiene. No architectural
changes; fits format v3 as-is.

## Decisions

1. **Target = `detailed_sport_info`**; the free-form `sport` field is ignored.
2. **`PARASPORTS_HAND_CYCLING → cycling`**, consistent with Wahoo's handcycling call.
3. **`platform_version = "AccessLink API v3 (fetched 2026-06-08)"`** — pin API version
   + fetch date, since the sport list is unversioned.
