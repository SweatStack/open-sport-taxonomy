# 015 â€” Platform-keyed mappings: format v3

## Background

External feedback flagged that `garmin_fit.yaml` (v0.4.0) is missing the indoor variants `indoor_cycling`, `indoor_rowing`, and `treadmill`. Investigation revealed the gap is a structural artifact of the current format direction, not a contributor mistake. Mapping files are keyed by OST sport string â€” an unbounded domain (any sport Ã— any modifier subset). The platform enum, the finite side, is referenced only indirectly. A contributor adds rows by asking "for this OST sport, what's the platform target?" and never structurally encounters `indoor_cycling`, because it isn't an OST sport.

This plan replaces the mapping format. Format v3 keys each mapping file by **platform target** â€” the finite, enumerable side. Every platform value gets exactly one row, annotated with its OST equivalent (or `null`). Coverage becomes structural: a row is required for every value in `reference/<platform>/targets.yaml`, enforced at load time. The 0.4.0 oversight cannot recur because every sub_sport in the FIT SDK reference must appear as a row before the file passes lint.

With zero production consumers, this is a clean breaking change. No compatibility shims.

## Design principles

1. **Key by the finite side.** OST is sparse and unbounded; platforms are dense and enumerable. The data structure reflects this.
2. **Coverage is structural.** Every value in the bundled reference enum must appear as a row. New SDK release â†’ new rows or load fails.
3. **Decode is a lookup; encode is a search.** Honest asymmetry. The two algorithms have different shapes because the data has different shapes.
4. **Runtime is data-driven.** No per-platform code in encode or decode. Coarsening rules, fallbacks, and encode preferences all live in YAML. (Tooling like `scaffold.py` may carry per-platform knowledge â€” that's a build-time concern, not a runtime one.)
5. **`null` is explicit.** "Considered, no OST equivalent" is textually distinct from "forgotten." Silent omission is a load error.
6. **Strict load.** Unknown keys, malformed sport strings, missing rows, duplicate `preferred` flags â€” all fail loudly at load time.

## Prior art

The platform-keyed shape is standard for finite-to-sparse translation tables:

- **ICD GEM**: keyed by ICD code (finite); each row gives the equivalent in the other code system.
- **Unicode normalization tables**: keyed by codepoint (finite); each row gives the canonical form.
- **SNOMED CT description files**: keyed by description ID (finite); each row points to a concept with one description marked preferred.
- **OSM deprecated-tag mappings**: keyed by the deprecated tag (finite); each row gives the replacement.

None of these key the table by the unbounded side. v3 follows the established pattern.

## Reference data contract

Each platform's `reference/<platform>/` directory must contain a canonical `targets.yaml` file enumerating every legal target value for that platform. Example for FIT:

```yaml
# reference/garmin-fit-sdk/targets.yaml
# Derived from sports.yaml Ã— sub_sports.yaml via the FIT SDK's associated_sports rule.
# Generated by scripts/build_reference.py (FIT-specific build step).
- { sport: 0, sub_sport: 0 }    # generic / generic
- { sport: 1, sub_sport: 0 }    # running / generic
- { sport: 1, sub_sport: 1 }    # running / treadmill
- { sport: 1, sub_sport: 2 }    # running / street
- { sport: 2, sub_sport: 0 }    # cycling / generic
- { sport: 2, sub_sport: 5 }    # cycling / spin
- { sport: 2, sub_sport: 6 }    # cycling / indoor_cycling
...
```

`targets.yaml` is **the** authoritative enumeration consumed by the loader, the linter, and the scaffolder. How it's produced is platform-specific â€” a small per-platform script under `scripts/build_reference/<platform>.py` derives it from the platform's source data (FIT's `sports.yaml Ã— sub_sports.yaml` with `associated_sports`; Strava's `sport_types.yaml`; HealthKit's `workout_activity_types.yaml` â€” each is a one-shot transformation). The runtime sees only the flat list; per-platform knowledge stays in the build step.

A `targets.yaml` change requires re-running the build script for that platform; CI runs the build and asserts `git diff --exit-code` to catch drift.

## Format specification

### File structure

```yaml
format_version: 3
platform: garmin_fit
platform_version: "FIT SDK 21.133"

fallback:
  encode: { sport: 0, sub_sport: 0 }
  decode: generic

target_coarsening:
  - reset: { sub_sport: 0 }

entries:
  - target: { sport: 0, sub_sport: 0 }    # generic / generic
    sport: generic
    preferred: true

  - target: { sport: 1, sub_sport: 0 }    # running / generic
    sport: running
    preferred: true

  - target: { sport: 1, sub_sport: 1 }    # running / treadmill
    sport: running+stationary
    preferred: true

  - target: { sport: 2, sub_sport: 5 }    # cycling / spin
    sport: cycling+stationary

  - target: { sport: 2, sub_sport: 6 }    # cycling / indoor_cycling
    sport: cycling+stationary
    preferred: true

  - target: { sport: 4, sub_sport: 1 }    # fitness_equipment / treadmill
    sport: running+stationary

  - target: { sport: 4, sub_sport: 14 }   # fitness_equipment / indoor_rowing
    sport: rowing+stationary
    preferred: true

  - target: { sport: 4, sub_sport: 15 }   # fitness_equipment / elliptical
    sport: null    # no OST equivalent â€” decodes to fallback.decode
```

Rows are sorted lexicographically by `target`. The file is scannable side-by-side with `reference/<platform>/targets.yaml`.

### Field reference

| Field | Required | Meaning |
|---|---|---|
| `format_version` | yes | Integer. Currently `3`. Loaders refuse other values. |
| `platform` | yes | Platform identifier. Must match the directory under `reference/`. |
| `platform_version` | yes | Version string of the platform spec bundled in `reference/`. |
| `fallback.encode` | yes | Platform value returned when no encode candidate matches. |
| `fallback.decode` | yes | OST sport string returned when no decode candidate matches. |
| `target_coarsening` | optional | List of rewrite rules used when decoding targets absent from `entries` (forward-compat with newer platform versions). |
| `entries` | yes | List of rows. Each row is one platform value with its OST annotation. |
| `entries[].target` | yes | Platform-specific value. Shape is platform-specific. |
| `entries[].sport` | yes | OST sport string (e.g. `cycling.road+stationary`) or `null` if no OST equivalent. |
| `entries[].preferred` | optional, default `false` | If `true`, this row is the encode target for its `sport`. Forbidden when `sport: null`. |

### Target coarsening

`target_coarsening` applies only when decoding a target that doesn't appear in `entries` â€” i.e., a value from a newer platform version than the bundled `targets.yaml`. Because rule 6 below requires every bundled target to have a row, coarsening is inactive for any in-snapshot target.

**Semantics.** Each rule independently produces one candidate target. Rules apply in declaration order; for each rule, the produced candidate is looked up in `entries_by_target`; the first hit wins. A rule whose output equals the input (no-op) is skipped without lookup. If no rule produces a hit, decode returns `fallback.decode`.

**Rules do NOT chain.** Rule N is applied to the original target, not to rule Nâˆ’1's output. This keeps each rule's effect locally inspectable.

**The `reset` rule** sets the listed fields to the values declared inline:

```yaml
target_coarsening:
  - reset: { sub_sport: 0 }                  # (2, 99) â†’ (2, 0)
  - reset: { sport: 0, sub_sport: 0 }        # (99, 99) â†’ (0, 0)
```

`reset` is the only rule kind defined in v3. Further rule kinds (e.g. `replace`, `parent_lookup`) require a `format_version` bump.

**Validation of `reset`.** Every field named in a `reset` rule must exist in the platform's target shape, else load fails. Fields not named in a rule pass through unchanged.

### Validation rules (load-time, strict)

1. `format_version == 3`.
2. `platform` matches a directory under `reference/`.
3. No unknown top-level or per-entry keys.
4. Every `target` is unique within `entries`.
5. Every `target` appears in `reference/<platform>/targets.yaml`.
6. Every value in `reference/<platform>/targets.yaml` has exactly one matching `target` row in `entries`. **This is the rule that makes the 0.4.0 oversight impossible.**
7. Every non-null `sport` parses as a valid sport string per `schema.yaml`. Sport codes must be **standard** (i.e., present in `schema.yaml`); non-standard codes are forbidden in mapping files. Modifiers must be sorted alphabetically in the canonical form.
8. For each non-null `sport` value appearing in `entries`, exactly one entry has `preferred: true`.
9. `preferred: true` is forbidden on rows where `sport: null`.
10. Round-trip on preferred entries: `decode(target) == sport` and `encode(sport) == target`.
11. Decode of non-preferred synonym rows: `decode(target_synonym) == sport_of_canonical_row`.
12. `fallback.decode` parses as a valid sport string AND equals the `sport` of some preferred entry in `entries` (guarantees the fallback round-trips to a real platform value via encode).
13. Every field named in a `reset` rule exists in the platform's target shape.

A load failure on any rule aborts; the library refuses to operate on a partially-valid mapping file.

## Algorithms

### Decode `target â†’ sport`

```
decode(target):
  for candidate in [target, *coarsening_candidates(target)]:
    if candidate == target and candidate already tried:    # skip no-op rules
      continue
    entry = entries_by_target.get(candidate)
    if entry is not None:
      return entry.sport if entry.sport is not None else fallback.decode
  return fallback.decode

coarsening_candidates(target):
  for rule in target_coarsening:
    candidate = apply(rule, target)
    if candidate != target:
      yield candidate
```

A direct lookup in the platform-keyed table. The coarsening walk handles only forward-compat (target newer than the bundled `targets.yaml`). `entry.sport == null` routes to `fallback.decode` rather than continuing the walk â€” the row's author asserted "no OST equivalent" and that assertion is respected.

### Encode `sport â†’ target`

```
encode(sport):
  for candidate in ost_hierarchy_walk(sport):
    entry = preferred_index.get(candidate)
    if entry is not None:
      return entry.target
  return fallback.encode

ost_hierarchy_walk(sport):
  # Modifiers dominate discipline depth.
  yield (sport.code, sport.modifiers)               # exact
  for ancestor in strict_ancestors(sport.code):     # nearest first
    yield (ancestor, sport.modifiers)
  if sport.modifiers:                               # only if there's a difference
    yield (sport.code, âˆ…)
    for ancestor in strict_ancestors(sport.code):
      yield (ancestor, âˆ…)
```

`preferred_index` is built at load time by selecting rows with `preferred == true` and inverting them: `(sport.code, sport.modifiers) â†’ target`.

**Rationale for modifiers-dominate ordering.** Modifiers like `stationary`, `virtual`, `assisted` represent measurement circumstances that materially change activity interpretation (indoor vs outdoor, electric vs human-powered). Discipline depth is finer-grained categorization *within* the same circumstances. Dropping `+stationary` to preserve `.road` produces a worse error (indoor trainer ride displayed as a 40 km outdoor route) than dropping `.road` to preserve `+stationary` (correctly indoor, less specific discipline).

This is a deliberate change from v1, which let discipline depth dominate.

### Why the asymmetry is correct

Decode's domain is finite (the platform enum). Lookup is the natural shape. One step plus optional forward-compat coarsening.

Encode's domain is unbounded (any OST sport Ã— any modifier subset). Hierarchy search is the natural shape. Multiple candidates ordered by priority.

Earlier drafts pursued symmetric algorithms. That was a contortion. The new asymmetry reflects the data's asymmetry.

## Tooling

Per principle 4, tooling may carry per-platform knowledge â€” it's a build-time concern.

### `scripts/lint.py`
Runs validation rules 1â€“13. CI invokes it on every PR. `--fix` mode sorts entries, normalizes modifier order in `sport:` values, and applies trivial formatting fixes.

### `scripts/scaffold.py <platform>`
Reads `reference/<platform>/targets.yaml` and generates a complete skeleton mapping file: one row per target, `sport: null` everywhere, no `preferred` flags. The contributor edits annotations. Reduces "add a new platform" from a blank-page exercise to a fill-in-the-table exercise.

### `scripts/build_reference/<platform>.py`
Platform-specific transformations that produce `reference/<platform>/targets.yaml` from the upstream source data. One per platform. The only place per-platform code is permitted, and it runs at commit time, never at runtime. CI runs all build scripts and asserts `git diff --exit-code` to catch drift.

## Decisions deferred to implementation

1. **`target` serialization shape.** FIT's `{sport: 2, sub_sport: 6}` is verbose. A compact form (`"2/6"`) saves characters but loses field clarity and breaks the side-by-side scan against `reference/garmin-fit-sdk/`. Default: keep the struct form unless a strong case emerges during migration.
2. **`virtual_activity (58)` mapping.** FIT's generic virtual flag has no clean OST equivalent (the OST `+virtual` modifier must attach to a specific sport). Most likely: `sport: null` with a comment, revisited when OST grows a generic-virtual-capable parent.
3. **Sort order for platforms with named targets** (Strava: `Ride`, `VirtualRide`, â€¦). Case-insensitive alphabetical. Trivial; confirm during Phase 3.
4. **Second coarsening rule on FIT.** The plan includes `{ reset: { sport: 0, sub_sport: 0 } }` as a backstop. Useful only for a future FIT SDK that adds entire new sport ids; not strictly required. Confirm during Phase 2 that the lookup-cost is negligible (it is â€” both candidates are constant-time).

## Out of scope

- **New OST sport codes** for FIT's `fitness_equipment` family (elliptical, stair_climbing, hand_cycle, etc.). Each is a separate taxonomy proposal against `schema.yaml`.
- **Class-2 ambiguities** (`backcountry` under both alpine_skiing and snowboarding; `whitewater` under kayaking and rafting). Each variant has a distinct `target` so they sit as distinct rows; resolving the OST side requires new sport codes in `schema.yaml`.
- **Reference-drift detection beyond build script idempotency** (e.g. comparing bundled reference against the live upstream API). Useful follow-up; not blocking.
- **Python API ergonomics.** Tracked separately in plan 016.
